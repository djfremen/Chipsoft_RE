# Wire opcode catalog — j2534_interface.dll

Synthesis built from `2026-05-05-ghidra-opcodes.md` (auto-generated by
`tools/ghidra_scripts/DumpOpcodes.java`). That file is the raw dump and is
re-runnable; **this** file is the human-curated table for use when writing
the Android client.

`FUN_1001d270` is `send_and_wait(this, opcode, msg, len, out, timeout_ms)`.
Every wrapper below is a 1-opcode dispatcher: it stamps the opcode literal,
hands the buffer to the IOCP write thread, and blocks on a reply.

## Key takeaways for the Android client

1. **24 distinct opcodes** in use, fully covering the J2534 v04.04 surface.
   Closes the count: the `2026-05-05-first-pass.md` note's "25+ opcodes
   identified from the response validator switch" was within rounding.

2. **PassThruWriteMsgs has two paths.**
   - `0x22` is the async path (the one labelled "Add async message to queue"
     in `ghidra-transport.md`); it goes through the IOCP queue and lets the
     caller specify a timeout that's added to a base of 3000 ms.
   - `0x0F` is a parallel **synchronous** wrapper (`FUN_10017f60`) that uses
     the same length expression but with no queueing.
   - For the Android client, only one path needs to be implemented;
     `0x22` matches the typical Tech2 usage observed in driver logs.

3. **PassThruOpen fires three opcodes in sequence** (`0x20`, `0x01`, `0x08`):
   most likely `reset → probe → arm`. All have length 0 (opcode-only with
   no payload); responses populate the device-id / version fields.

4. **PassThruClose fires `0x20`** — same opcode as Open's reset step.
   Reasonable: open prepares the channel by tearing down anything stale,
   close uses the same teardown.

5. **PassThruConnect and PassThruGetLastError fire NO wire opcode.**
   - `Connect` allocates the host-side channel object and stores parameters
     (protocol id, baud, flags); the actual bus init happens via subsequent
     `PassThruIoctl` calls.
   - `GetLastError` reads a host-side error string. No device round-trip.
   - Implication: the Android client must do the same — Connect is a pure
     local state operation, no wire traffic.

6. **PassThruIoctl is the catch-all dispatch** — 11 distinct opcodes
   (`0x06, 0x07, 0x0B, 0x0C, 0x0D, 0x11, 0x12, 0x18, 0x1C, 0x21, 0x23`)
   reached through the `param_2` switch in `FUN_100070a0`. Mapping each
   PassThru-Ioctl id (1, 2, 3, 4, 5, 7, 8, 9, 0xA, 0xE) to its opcode is the
   next-finer level of detail; the call snippets in
   `2026-05-05-ghidra-opcodes.md` plus the switch in
   `2026-05-05-ghidra-j2534.md` give it directly.

7. **Length field is mostly constant.** Only opcodes `0x06`, `0x0F`, and
   `0x22` have dynamic length (`(short)param[N] + offset`). For everything
   else, the wire packet is fully predictable from the opcode alone.

8. **Timeout is always `3000` ms** for non-write opcodes; for `0x0F` and
   `0x22` it's `*param_2 + 3000` (caller-specified plus a 3 s slack).
   The Android client should use the same convention.

## What's still open

- **PassThruIoctl id → opcode mapping.** We know which 11 opcodes Ioctl
  uses, but the exact mapping from J2534 Ioctl id (1=GET_CONFIG, 2=SET_CONFIG,
  3=READ_VBATT, 4=FIVE_BAUD_INIT, 5=FAST_INIT, 7=CLEAR_TX_BUFFER, 8=CLEAR_RX_BUFFER,
  9=CLEAR_PERIODIC_MSGS, 10=CLEAR_MSG_FILTERS, 14=READ_PROG_VOLTAGE) to wire
  opcode is implicit in the switch at `FUN_100070a0`. Looking at the switch
  body decompile in `notes/2026-05-05-ghidra-j2534.md`, it's recoverable by
  reading the case bodies — each case calls one of the wrappers above.
  Five-minute job, lower priority than building the client skeleton.

- **Reply framing.** The opcode goes in the request envelope; the reply has
  its own envelope (`(R) << ` log marker in `FUN_1001c810`). Whether the
  reply opcode mirrors the request opcode or uses a separate ack code is
  not yet confirmed. Bench capture answers this trivially.

- **Opcode `0xF012`** mentioned in the audit synthesis: not seen in any
  direct call site. Either it's an internal sub-opcode in a payload, or it
  comes from `CSTech2Win.dll` which has different transport. Worth a
  follow-up grep against the j2534 binary for `f012` constants.
